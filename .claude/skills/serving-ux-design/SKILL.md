# Serving & UX Design (Stage 4)

## 언제 사용하는가
데모 서빙서버와 사용자/관리자 화면, 로깅·시각화를 설계할 때. `serving-ux-architect`의 주 스킬.

## 워크플로우

### 1. 서빙 아키텍처
기본값: **FastAPI**(추론/온톨로지 API) + **Streamlit**(데모 화면). 정의할 것:
- 엔드포인트: `POST /classify`(질문→추론), `POST /answer`(추가질문 응답), `GET /logs/{session}`(관리자), `GET /ontology/{code}`
- 요청/응답 스키마(JSON)
- 화면-서버-온톨로지 연동도(Mermaid)

### 2. 사용자 화면
- 흐름: HS2류 선택 + 품목 질문 입력 → cycle 진행 표시(단위별 후보군·추가질문) → 최종 HS10 + **단위별 근거** 표시.
- 대화형으로 추가 질문에 답하면 다음 단위로 진행되는 UX.

### 3. 관리자 화면 (로깅·시각화)
agentic-flow-design의 로그를 기반으로 시각화한다:
- **추론 트리**: HS2→10 경로와 각 단위 후보군
- **확신도 차트**: 단위별 score 막대/추이
- **후보군 변화**: 단위별 후보 목록과 탈락 사유
- **질문-판별 추적**: 사용자 질문 ↔ 어떤 판별에 기여했는지
목적: 근거·증적·확률을 쉽게 파악.

### 4. 로깅 스키마 (JSON)
시각화가 소비할 구조를 확정한다. 예:
```json
{
  "session_id": "...", "query": "...", "hs2": "85",
  "steps": [
    {"level": "HS4", "candidates": [{"code":"8471","score":0.7,"reason":"..."}],
     "question": "...", "chosen": "8471", "evidence": ["note:...","ruling:..."]}
  ],
  "final": {"hs10": "8471.30.0000", "rationale_by_level": {...}}
}
```

### 5. 출력
`_workspace/design_serving.md`에 저장.

## 품질 기준
- 엔드포인트·스키마가 app-engineer가 바로 구현 가능한 수준
- 사용자/관리자 화면 모두 와이어프레임/흐름이 명세됨
- 로깅 스키마가 agentic-flow-design의 로그 필드와 정합

## 흔한 실수
- 관리자 화면을 단순 로그 덤프로 설계 — 근거/확신도가 한눈에 안 보임
- 로그 스키마를 화면 요구와 따로 정의해 필드 불일치 발생
