# Lab Documentation (단계별 문서 + Stage 5 최종보고서)

## 언제 사용하는가
각 단계 완료 시 산출물 문서를 쓰고, 전 단계 종료 후 최종보고서를 작성할 때. `lab-documenter`의 주 스킬.

## 작성 원칙
- **한국어 비즈니스 톤**: 격식체("~합니다"). 영문 용어 첫 등장 시 한국어 병기.
- **표·다이어그램 우선**: 비교/구조는 markdown 표·Mermaid로. 본문 나열 금지.
- **부분 수정 안전**: 후속 수정 시 지정 섹션만 고쳐 git diff가 깔끔하게.

## A. 단계별 문서화 (`docs/stage{N}_*.md`)
각 단계 완료 직후 해당 `_workspace/` 산출물을 읽고 정리한다:
- `docs/stage1_research.md` — 리서치 요약(3-layer/Agentic 패턴/기획 시사점)
- `docs/stage2_analysis.md` — 포털 분석/데이터 relation/엔진 선정
- `docs/stage3_requirements.md` — 요구사항정의서·WBS·작업지시서 요약 + 사용자 확정 결정
- `docs/stage4_design.md` — 온톨로지·Agentic flow·서빙/UX 통합 설계 요약
- `docs/stage5_mvp.md` — 구축 결과/데모 실행법/검수 결과
각 문서: 목적 → 핵심 표/다이어그램 → 결정 사항 → 산출물 위치.

## B. 최종보고서 (`reports/final_report.md`, Stage 6)
```markdown
# HS 온톨로지 실험실 — 최종보고서
## 1. Executive Summary
(반 페이지. 무엇을 만들었고 무엇을 입증했는가)
## 2. 작업과정 요약
(리서치→분석→설계→MVP 흐름, 단계별 핵심 결정)
## 3. 성과 및 산출물
(단계별 산출물 표 + 위치, MVP 데모 구성)
## 4. 데모 결과
(시나리오·검수Gate 결과·평가지표)
## 5. 작업의 의의
## 6. 향후 확장 고려사항 (HS 97류 전체)
(데이터/추론/서빙 확장 시 이슈, 비용, 단계적 로드맵)
## 부록: 산출물·출처 목록
```

## 품질 기준
- Executive Summary만 읽어도 결론 파악 가능
- 단계별 산출물 위치가 표로 정리됨
- 향후 확장 고려사항이 구체적(추측이 아니라 근거 기반)

## 흔한 실수
- 각 단계 문서를 안 만들고 최종보고서만 작성 — 단계별 추적성 상실
- 성과를 텍스트로만 풀어써 산출물 위치를 못 찾음
- 확장 고려사항을 일반론으로 채움
