# Agentic Flow & Evaluation Design (Stage 4)

## 언제 사용하는가
온톨로지 데이터 모델 위에서 동작하는 Agentic HS 추론 flow와 평가·검수 체계를 설계할 때. `agentic-flow-architect`의 주 스킬.

## 워크플로우

### 1. 데모 시나리오 확정
사용자가 특정 HS2류 내에서 질문 → HS2→4→6→10 각 단위에서 판단 → 최종 HS10 추천 + 단위별 근거 제시. 이 시나리오를 추론 flow의 기준으로 삼는다.

### 2. 추론 flow 단계 명세
각 단위(HS2/4/6/10)에 대해 정의한다:
- **후보군 도출**: 온톨로지에서 하위 후보 코드를 조회(Link 탐색) + LLM이 품목설명과 대조
- **추가 정보 요청**: 후보를 가를 수 없으면 사용자에게 변별 질문 생성
- **판단 기준**: 후보 선택 규칙(GRI 연동), 확신도(score) 산출
- **종료 조건**: HS10 도달 또는 확신도 임계치
- **근거 기록**: 단위별 선택 근거·증적(참조한 노드/규정)·확신도 저장
Mermaid 다이어그램으로 cycle을 표현한다.

### 3. 온톨로지 호출 인터페이스
추론이 ontology-architect의 Object/Link/Action을 어떻게 호출하는지 함수 인터페이스로 명세한다(예: `get_children(code)`, `apply_gri(candidates, desc)`).

### 4. 프레임워크 권장
기본값: **Claude API tool use**(Anthropic SDK, 프롬프트 캐싱). 대안: LangGraph(상태 그래프 cycle). 리서치·분석 근거로 권장안과 사유를 제시한다.

### 5. 로그 구조 (시각화 입력)
단위별 근거·증적·확신도를 담는 JSON 로그 필드를 정의한다 — serving-ux-architect의 시각화가 소비.

### 6. 검수Gate & 평가 방법론
- **단계별 검수Gate**: 수집/정제/온톨로지/추론/서빙 각 단계의 합·불 판정 기준과 점검 방법.
- **평가지표**: 정확도@HS6/@HS10, 근거 충실도, 평균 cycle 수, 추가질문 적정성 등. 측정 방법과 골드셋 구성 방안 포함.

### 7. 출력
`_workspace/design_agentic.md`에 저장.

## 품질 기준
- 각 단위의 후보 도출/종료 조건/근거 기록이 구현 가능 수준으로 명세
- 온톨로지 호출 인터페이스가 ontology 설계와 정합
- 검수Gate 기준이 PASS/FAIL 판정 가능하게 구체적

## 흔한 실수
- cycle 종료 조건·실패(분류 불가) 처리를 빠뜨림
- 근거·확신도 기록 구조를 정의하지 않아 관리자 시각화가 불가능
- 평가지표를 측정 방법 없이 나열만 함
