# Requirements Definition (Stage 3)

## 언제 사용하는가
분석 결과를 바탕으로 요구사항을 체계화하고, 후속 단계의 WBS·작업지시서를 만들 때. `requirements-analyst`의 주 스킬.

## 워크플로우

### 1. 요구사항 수집·구조화
`analysis.md`와 `README.md`를 읽고, 흩어진 요구사항을 기능(FR)/비기능(NFR)으로 분류한다. README의 각 단계 항목(서빙서버, 사용자/관리자 화면, 로깅·시각화, 검수Gate, 평가 방법론, MVP 데모 등)을 빠짐없이 요구사항으로 변환하고 ID를 부여한다.

```markdown
| ID | 유형 | 요구사항 | 출처 | 우선순위 |
|----|------|---------|------|---------|
| FR-1 | 기능 | 사용자가 HS2류 내 질문으로 HS10까지 분류받는다 | README 4 | High |
| NFR-1 | 로깅 | 단위별 근거·증적·확신도를 구조화 로깅 | README 4 | High |
```

### 2. 결정 필요 항목 → 후보군 제시 (핵심)
정답이 하나가 아닌 항목은 단정하지 말고, **결정 필요 항목 + 2~4개 후보군 + 트레이드오프**로 정리한다. 이 목록을 오케스트레이터가 사용자에게 제시(AskUserQuestion)하고, 결정 결과를 받아 확정한다.

```markdown
## 사용자 결정 필요 항목
### D-1. MVP 대상 HS2류
- 후보 A: 85류(전기기기) — 데이터 풍부, 사례 많음 / 후보군 큼
- 후보 B: 09류(커피·향신료) — 단순, 데모 명료 / 변별 빈약
- 트레이드오프: ...
```
사용자 결정이 오면 "확정 결정 기록"에 옮긴다.

### 3. WBS 작성
요구사항 ID를 참조하는 작업 분해 구조를 만든다.
```markdown
| 작업ID | 작업범위 | 담당 에이전트 | 산출물 | 의존 | 완료Gate | 일정 | 완료 |
|--------|---------|--------------|--------|------|---------|------|------|
| W-1 | 온톨로지 데이터 모델 설계 | ontology-architect | design_ontology.md | - | 스키마 구현가능성 검토 | D+1 | ☐ |
| W-5 | 데이터 수집·정제·온톨로지 구축 | data-engineer | mvp/ontology | W-1 | 계층 무결성·재현 | D+3 | ☐ |
```

### 4. 작업지시서 (서브Agent별)
후속 단계 각 에이전트가 독립 수행할 수 있도록 지시서를 작성한다:
- **작업 내용**: 무엇을 만드는가(요구사항 ID 참조)
- **입력**: 어떤 산출물을 읽는가
- **검수방법**: 어떻게 확인하는가
- **완료Gate**: PASS 조건(측정 가능하게)

### 5. 출력
`_workspace/requirements.md`(요구사항정의서) + `_workspace/wbs.md`(WBS + 작업지시서)에 저장.

## 품질 기준
- README의 모든 단계 항목이 요구사항으로 추적됨(누락 0)
- 결정 필요 항목이 후보군+트레이드오프로 제시되어 사용자가 고르기만 하면 됨
- WBS의 모든 작업에 담당·산출물·완료Gate·의존이 명시됨

## 흔한 실수
- 애매한 항목을 임의로 단정 — 후보군으로 제시해 사용자 결정에 맡겨야 함
- 요구사항을 나열만 하고 WBS/작업지시서로 연결하지 않음
- 완료Gate가 측정 불가능한 추상 표현
