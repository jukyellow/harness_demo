# Final Report Writing

## 언제 사용하는가
`_workspace/` 산출물 3개를 통합하여 최종 보고서를 작성할 때. `report-writer` 에이전트의 주 스킬.

## 워크플로우

### 1. 입력 통합
다음 파일을 모두 읽는다:
- `_workspace/01_palantir_research.md`
- `_workspace/02_hscode_analysis.md`
- `_workspace/03_ontology_design.md`

누락 시 오케스트레이터에 보고하고, 가능하면 부분 입력으로 진행한다.

### 2. 보고서 구조 (필수)

```markdown
# HS코드 온톨로지 적용 방안 — 팔란티어 3-layer 패턴 기반

**작성일**: YYYY-MM-DD
**버전**: 1.0

## 1. Executive Summary
(반 페이지 분량. 핵심 결론과 권고를 먼저 제시. 의사결정자가 이 섹션만 읽어도 결론 파악 가능해야 함)

## 2. 배경 및 목적
- 왜 온톨로지인가
- 왜 HS코드인가
- 본 보고서의 범위

## 3. 팔란티어 3-layer 온톨로지 개요
- Object / Link / Action 각 레이어 정의
- 통합 패턴 다이어그램(Mermaid)

## 4. HS코드 도메인 분석
- 계층 구조 (표)
- 핵심 엔티티 및 관계
- 운영 프로세스 요약

## 5. 적용 대안 비교
| 비교 축 | 대안 A | 대안 B | (대안 C) |
|---------|--------|--------|----------|
| 전략 | ... | ... | ... |
| 세분도 | ... | ... | ... |
| 도입 난이도 | ... | ... | ... |
| 권장 시나리오 | ... | ... | ... |

## 6. 권고안 및 도입 로드맵
- 권고 대안 및 근거
- 단계별 도입 계획 (MVP → 확장)
- 위험 요소 및 대응

## 부록 A: 상세 매핑 명세
(설계안 원본의 Object/Link/Action 표 전체)

## 부록 B: 출처 목록
- 팔란티어 자료
- HS코드/WCO/관세청 자료
```

### 3. 작성 원칙
- **한국어 비즈니스 톤**: "~합니다", "~로 판단됩니다" 같은 격식체 사용. 기술 용어는 한국어와 영어 병기 (첫 등장 시).
- **표 우선**: 비교·열거는 markdown table로 정리한다. 본문 텍스트로 풀어쓰지 말 것.
- **다이어그램**: Object/Link 관계나 워크플로우는 Mermaid 다이어그램으로 표현.
  ```mermaid
  graph LR
    Commodity --classified_as--> HSCodeItem
    HSCodeItem --parent_of--> HSCodeItem
  ```
- **인용 절제**: 본문에는 핵심 인용 1~2개만, 나머지 출처는 부록 B로.

### 4. 출력
`reports/hscode-ontology-report.md`에 저장한다. `reports/` 디렉토리가 없으면 생성한다.

### 5. 후속 수정 시
이전 보고서가 있으면 읽고, 사용자가 지정한 섹션만 수정한다. 변경 부분은 git diff로 확인 가능해야 하므로 무관한 섹션은 건드리지 않는다.

## 품질 기준
- Executive Summary가 보고서 단독 요약으로 충분 (다른 섹션 안 읽어도 결론 파악 가능)
- 적용 대안 비교가 표로 정리되어 1페이지 내 비교 가능
- 모든 영문 약어 첫 등장 시 한국어 병기 (예: HS(통일상품분류체계))

## 흔한 실수
- Executive Summary에 결론을 미루고 배경만 쓰는 것 — 의사결정자는 결론을 먼저 본다
- 적용 대안을 본문 텍스트로 풀어쓰는 것 — 표로 정리하지 않으면 비교 불가
- 부록 없이 본문에 모든 출처 인용 삽입 — 가독성 저하
