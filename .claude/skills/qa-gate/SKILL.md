# QA Gate (Stage 4 검토 + Stage 5 단계별 검수)

## 언제 사용하는가
MVP 구축 파이프라인의 각 단계 합·불을 판정하고 데모를 실제 구동해 검증할 때. `qa-gate`의 주 스킬.

## 핵심 원칙
- **포그라운드 + 실행**: 검증 스크립트를 실행하고 리포트를 쓰므로 포그라운드 `general-purpose`로 동작.
- **존재가 아니라 동작 확인**: 파일이 있는지가 아니라 실제로 돌려서 결과를 본다.
- **점진적 검수**: 각 단계 완성 직후 검수(incremental). 전체 후 1회 금지.

## 워크플로우

### 1. Gate 기준 로드
`design_agentic.md`의 단계별 검수Gate 기준·평가지표를 읽는다.

### 2. 단계별 검수 (실행 기반)
| 단계 | 점검 예시 |
|------|----------|
| 수집 | raw 데이터 존재·필드 충족, 폴백 시 가정 명시 여부 |
| 정제 | 계층 무결성(고아 노드 0), 중복/결측 처리 |
| 온톨로지 | `build` 재현, 노드/엣지 수, 인터페이스 함수 동작 |
| 서빙 | FastAPI 기동, 엔드포인트 응답 shape ↔ 설계 스키마 일치 |
| 화면 | Streamlit 기동, 사용자/관리자 화면 렌더, 로그 필드 ↔ 시각화 필드 일치 |

### 3. 경계면 교차 비교
- 온톨로지 스키마 ↔ 적재 결과
- FastAPI 응답 ↔ Streamlit 소비 코드
- 추론 로그 스키마 ↔ 관리자 시각화가 읽는 필드
불일치는 구체적 위치와 함께 FAIL 처리.

### 4. 데모 실행 검증
데모 시나리오(특정 HS2류 질문 → cycle → HS10 추천 + 단위별 근거)를 실제 실행한다. 키 부재 시 mock 경로로 기능 동작 확인 + "실 추론 미검증" 명시.

### 5. 평가지표 측정
가능한 지표(정확도@HS6/@HS10, 평균 cycle 수 등)를 골드셋/샘플로 측정한다.

### 6. 판정 & 출력
각 Gate에 PASS/FAIL + 사유 + 재작업 항목. FAIL은 담당 엔지니어(`data-engineer`/`app-engineer`)에 구체적 수정 요청. `_workspace/qa_gate.md`에 표로 기록.

## 품질 기준
- 모든 PASS 판정에 실행 근거(명령·결과)가 있음
- FAIL에 재현 방법과 수정 대상이 명시됨

## 흔한 실수
- 코드를 읽고 "맞을 것 같다"로 PASS — 반드시 실행
- 경계면(응답 shape, 로그 필드) 교차 비교 누락
- 환경 문제를 제품 결함과 혼동
